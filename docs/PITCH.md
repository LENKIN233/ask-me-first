# 🎯 Ask My Avatar First — 先问我的分身

## 一个将"个人工作沟通层"接口化的智能数字分身系统

> **参赛项目说明书 v2.0**

---

## 一、首页摘要

### 🔑 一句话

**让别人先接触我的分身，而不是先打断我本人。**

### 项目简介

**Ask My Avatar First** 是一套完整的个人工作数字分身系统，基于 OpenClaw Agent 框架深度定制开发。

它不是一个通用聊天机器人，不是人格模拟器，也不是知识库 Q&A 工具——它是一个**真正运行在生产环境中的"个人工作接口层"**：当同事、合作方或管理者想找你时，他们首先接触的是你的数字分身。分身实时感知你的工作状态，根据来访者身份和问题性质进行三级智能决策——**直接回答、部分回答、或升级给本人**。

### 它解决什么问题

在真实工作中，**高价值个体最大的隐性成本不是工作量，而是被打断的成本。**

研究表明，一次上下文切换平均需要 23 分钟恢复专注。但大量日常沟通并不需要本人即刻介入，它们只是想知道：

- 你现在忙吗？适不适合打扰？
- 事情推进到哪了？有没有风险？
- 这个问题需不需要你亲自回答？

**今天的一切都在被 API 化，但工作中的"人"仍然只能靠直接打断来参与协作。**

Ask My Avatar First 给个人建立了一个全新的接口层：

> **把一个人在工作中可代理的沟通层，重构为一个有状态感知、有权限控制、有升级决策能力的数字接口。**

### 当前状态

✅ **已在飞书生产环境稳定运行**，作为真实工作中的第一接触面处理日常沟通。

## 二、项目核心亮点

### 💡 亮点 1：不是"另一个 AI 助手"——而是全新产品范式

大多数 AI 产品在比拼"更聪明的助手"。
**本项目在做的是一件更底层的事情：重新定义"别人怎么接触我"。**

它不是让 AI 替你做所有工作，而是让 AI 先替你**承接**别人对你的工作沟通。这使它从"工具"变成了"入口"——从"效率工具"跃迁到**"个人基础设施"**。

> 我们提出一个新概念：**Personal-as-an-Interface（PaaI）—— 个人即接口**

---

### 💡 亮点 2：解决了一个高频、真实、普遍存在且长期无人解决的问题

在任何组织中，这些场景每天都在发生：

| 场景 | 传统方式 | Ask My Avatar First |
|------|---------|-------------------|
| 同事问进度 | 直接打断你 → 上下文切换 23 分钟 | 分身实时同步状态，直接回答 |
| 合作方想了解背景 | 等你回消息 → 可能数小时 | 分身秒级响应，提供授权范围内的背景 |
| 管理者询问风险 | 又要开一个会 → 30 分钟 | 分身汇报已知信息，真正需要时才升级 |
| 对方不确定该不该打扰 | 犹豫 → 要么打断你，要么延误 | 分身明确告知当前可中断度 |

这些场景**极其高频**，但长期没有合适的产品形态来承接。本项目直接切入这个空白区域。

---

### 💡 亮点 3：有清晰边界的智能代理——不假装无所不能

与市面上"什么都试图回答"的聊天机器人不同，本系统有**精确的能力边界**：

**✅ 能做的：**
- 回答背景和已知进展
- 同步当前工作状态（在线/忙碌/专注/离线）
- 提供项目上下文和风险提示
- 判断是否应该找本人
- 根据权限提供分级信息

**🚫 不做的：**
- 最终拍板或做出承诺
- 越权泄露敏感信息
- 代替本人进行决策
- 伪装成本人

这种**边界感**是它区别于普通 AI 聊天机器人的核心——它不是在模拟人，而是在**代理人的工作沟通层**。

---

### 💡 亮点 4：多维实时状态感知——不是"虚的人格"，而是"真的工作态"

系统通过**多源数据融合**实时感知本人工作状态：

```
┌─────────────────────────────────────────────────┐
│              多源状态融合引擎                       │
├──────────────┬──────────────┬───────────────────┤
│  本地活跃度    │  飞书日历     │  显式声明          │
│  (Windows API) │  (Open API)  │  (/avatar 命令)   │
├──────────────┴──────────────┴───────────────────┤
│  置信度加权平均 → 冲突消解 → 统一状态输出          │
│  confidence: 0-1 │ interruptibility: 0-1         │
└─────────────────────────────────────────────────┘
```

- **本地活跃度检测**：通过 Windows API（GetForegroundWindow / Get-Process）实时识别前台应用，自动推断工作模式（coding / meeting / writing / browsing）
- **飞书日历集成**：读取未来 1 小时日历事件，判断是否在会议中
- **显式声明优先**：本人可通过 `/avatar` 命令显式覆盖
- **多源冲突消解**：当不同数据源给出矛盾判断时，自动降低置信度，触发 Partial 决策

分身表达的不是"我像什么"，而是**"我当前处于什么工作状态"**——这才是工作场景中真正有用的信息。

---

### 💡 亮点 5：完整的安全与权限体系——插件原生防御 + 四级信息可见性

#### 纯插件安全架构

| 层级 | 防御机制 | 作用 |
|------|---------|------|
| **Hook 层** | `agent:bootstrap` 钩子 + 身份注入 | 在 Agent 启动时注入用户角色约束和受限模式提示词 |
| **事件层** | `message_received` 事件监听 | 实时追踪会话身份映射，维护动态信任分数 |
| **上下文层** | `before_prompt_build` 事件 | 将当前 Avatar 状态自动注入 Prompt 上下文 |

#### 四级信息可见性控制

```
owner_only  → 仅本人可见（薪资、私人决策等）
trusted     → 高信任协作者可见（项目核心策略）
internal    → 团队成员可见（进度、背景、风险）
public      → 所有人可见（基础状态、通用信息）
```

**同一个问题，面对不同身份的人，分身的回答深度自动不同。** 这不是简单的"能答/不能答"二分法，而是**连续的信息可见性梯度**。

---

### 💡 亮点 6：动态信任评分——关系不是静态的

系统维护每个用户的动态信任分数（trustScore: 0-1），并根据交互行为实时调整：

- 本人确认了分身的回答 → 信任分 +0.05
- 正常交互 → 信任分 +0.01
- 长期无交互 → 信任分自然衰减
- 信任分影响：可获取的信息层级、是否需要升级确认、回答的详细程度

这意味着系统会随着使用**越来越准确**，而不是永远停在初始配置。

## 三、技术架构

### 3.1 系统总体架构

```
飞书消息 → OpenClaw Channel → Gateway(访问控制) → AvatarController → 决策引擎 → 模型生成 → 飞书回复

核心控制层：
┌──────────────────────────────────────────────────────────────────────┐
│                         AvatarController                             │
│                    （分身大脑 — 全链路编排器）                           │
├────────────────┬────────────────┬─────────────────┬─────────────────┤
│  StateDetector │ IdentityResolver│ EscalationRouter│ ReplyFormatter  │
│  状态感知引擎   │ 身份关系系统     │ 升级决策引擎     │ 回复生成器       │
├────────────────┴────────────────┴─────────────────┴─────────────────┤
│                         支撑工具层                                    │
│  CalendarTool  │  PresenceTool  │  ContextTool   │  MemoryTool      │
│  飞书日历       │  本地活跃度     │  项目上下文      │  长期记忆        │
└──────────────────────────────────────────────────────────────────────┘
```

**核心设计哲学**：模型不是"唯一大脑"，而是被状态、身份、权限、升级规则**约束后的表达器**。真正的智能在中间的决策层，而不是末端的生成层。

---

### 3.2 六大核心模块详解

#### 模块 1：AvatarController — 分身控制中枢

全链路编排器，接收消息后依次完成：身份解析 → 状态获取 → 升级决策 → 信息过滤 → 回复生成。

```typescript
async process(inbound): Promise<ProcessResult> {
  const identity  = await this.identityResolver.resolve(inbound.senderId);  // 你是谁
  const state     = await this.stateDetector.getState();                    // 我在干嘛
  const decision  = this.escalationRouter.decide(msg, identity, state);     // 该怎么处理
  const reply     = await this.generateReply(decision, state, identity);    // 怎么回答
  return { reply, decision, state, identity };
}
```

每次交互产生完整的**决策审计记录**（身份、状态、决策级别、原因），记录到 `queries.json`，支持后续分析和规则调优。

---

#### 模块 2：StateDetector — 多源状态融合引擎

**创新点**：不依赖单一数据源，而是将多个异构信号源**通过置信度加权平均进行融合**。

| 信号源 | 数据来源 | 检测内容 | 置信度权重 |
|-------|---------|---------|-----------|
| 本地活跃度 | Windows API (P/Invoke) | 前台窗口、进程名 → 推断工作模式 | 0.7 |
| 飞书日历 | Feishu Open API | 当前/即将开始的会议 | 0.9（会议中）|
| 显式声明 | /avatar 命令 | 本人主动设置 | 1.0（最高优先级）|

**输出状态模型**：
```typescript
interface AppState {
  availability: 'online' | 'busy' | 'focus' | 'offline';  // 可用性
  interruptibility: number;    // 可打断度 0-1（越高越适合打扰）
  current_mode: string;        // 工作模式：coding / meeting / writing / planning
  confidence: number;          // 状态可靠度 0-1（多源冲突时自动降低）
  evidence: Evidence[];        // 判断依据链（可解释 AI）
}
```

**关键特性**：
- 内置**TTL 缓存层**，避免高频检测造成性能开销
- 多源冲突时**自动降低置信度**，触发保守决策（Partial）
- 状态附带 Evidence 证据链，**回复中可向用户解释"为什么这么判断"**

---

#### 模块 3：IdentityResolver — 身份与关系引擎

不仅识别"你是谁"，还建模"你和我是什么关系"：

```json
{
  "userId": "ou_xxx",
  "identity": "member",           // 基础身份：admin / member / guest
  "infoLevel": "internal",        // 信息可见性级别
  "relationship": {
    "team": "backend",            // 所属团队
    "project": ["avatar"],        // 参与项目
    "role": "peer",               // 关系角色
    "trustScore": 0.7,            // 动态信任分（0-1）
    "lastInteraction": "2026-..."  // 最后交互时间
  }
}
```

**同一句"现在进展怎么样"**，来自同事、管理者、外部合作方，分身的回答策略完全不同——这是真正的**上下文感知**。

---

#### 模块 4：EscalationRouter — 三级升级决策引擎

系统的核心智能——将每个请求分流为三个级别：

| 级别 | 含义 | 触发条件 | 响应策略 |
|------|------|---------|---------|
| **Answer** | 可直接回答 | 高置信度 + 权限匹配 + 非敏感 | 完整回答 + 状态同步 |
| **Partial** | 部分回答 | 置信度不足 / 权限不足 / 需确认 | 提供背景 + 建议联系本人 |
| **Escalate** | 必须升级 | 敏感话题 / 承诺请求 / 显式升级 | 转交通知 + 原因说明 |

**内置 5 条默认规则 + 可热加载的自定义规则引擎**：

```
优先级 100: 显式升级 → "找本人"/"转接"/"升级" → Escalate
优先级  90: 敏感话题 → 预算/薪资/法律/承诺 → Escalate
优先级  85: 时间承诺 → "什么时候完成"/"deadline" → Escalate
优先级  80: 低置信度 → state.confidence < 0.6 → Partial
优先级  75: 权限不足 → guest 访问 internal 信息 → Partial
```

规则支持**关键词匹配 + JS 表达式条件**，无需改代码即可动态调整决策逻辑。

---

#### 模块 5：Permissions — 四级信息可见性策略

```
访问控制矩阵：
              public    internal    trusted    owner_only
  guest         ✅         ❌          ❌          ❌
  member        ✅         ✅          ❌          ❌
  trusted       ✅         ✅          ✅          ❌
  admin         ✅         ✅          ✅          ✅
```

分身在回答前，不只判断"能不能答"，还判断**"对这个人，能答到什么深度"**。超出权限的信息不会被注入到回复上下文中，从根源上杜绝信息泄露。

---

#### 模块 6：ReplyFormatter — 模板化回复生成器

使用预定义模板约束模型输出，确保回复风格一致、边界清晰：

```
Answer 模板:  "根据当前状态（{{state}}），{{answer}}"
Partial 模板: "当前：{{state}}。关于{{topic}}，仅能提供背景：{{context}}。建议@我本人确认。"
Escalate 模板: "已升级给本人，他将尽快回复。（原因：{{reason}}）"
```

**设计理念**：模型的职责是组织语言，不是决定边界。边界由上游的决策引擎严格控制。

---

### 3.3 纯插件架构 — 零侵入式全生命周期管理

v2.0 采用 **纯插件架构**，所有功能通过 OpenClaw Plugin API 实现，无需外部 Hook 目录或 Gateway 补丁：

```
Plugin Entry (index.ts) 注册的全部能力：

┌──────────────────────────────────────────────────────────────────┐
│  1. /avatar 命令     → registerCommand()                         │
│     读取/设置 Avatar 状态，管理员可手动覆盖                         │
│                                                                  │
│  2. message_received → api.on('message_received')                │
│     实时追踪会话身份 + 更新信任分数                                 │
│                                                                  │
│  3. agent:bootstrap  → api.registerHook(['agent:bootstrap'])     │
│     注入身份文件 + 受限模式提示词 + 初始化 AvatarController         │
│                                                                  │
│  4. before_prompt_build → api.on('before_prompt_build')          │
│     将 Avatar 实时状态注入 Prompt 上下文                           │
│                                                                  │
│  5. 后台服务          → registerService()                         │
│     每 10 分钟刷新状态 + 每小时衰减信任分                          │
└──────────────────────────────────────────────────────────────────┘
```

**设计优势**：
- **一键安装**：无需手动注入或修补框架文件
- **安全升级**：框架升级不会破坏插件功能
- **可移植性**：任意 OpenClaw 实例均可直接加载

## 四、商业价值与市场前景

### 4.1 市场洞察：一个被忽视的万亿级痛点

根据 Asana 2024 年《工作解剖学》报告：

- 知识工作者每天平均被打断 **56 次**
- **每次打断后恢复专注平均需要 23 分钟**（UC Irvine 研究）
- 60% 的工作时间花在"关于工作的沟通"上，而非工作本身
- 管理者每周花 **7.8 小时**在不必要的状态同步会议上

**高价值个体的真正稀缺资源不是知识或技能，而是：注意力、中断成本、决策时刻、上下文切换成本。**

现有的所有产品——Slack、飞书、Teams——都在让"打断你"变得更容易。
**没有任何产品在保护"不被打断"这个需求。**

Ask My Avatar First 填补了这个空白。

---

### 4.2 用户价值矩阵

| 价值维度 | 效果 |
|---------|------|

| 提高响应效率 | 简单问题秒级响应，无需等本人回来 |
| 建立工作边界 | 清晰的"能答/不能答"边界，减少模糊沟通 |
| 保护深度工作 | 专注模式下分身自动代理，本人不被打扰 |
| 信息安全 | 四级可见性控制，不担心信息泄露 |

#### 对组织

| 价值维度 | 效果 |
|---------|------|
| 减少会议 | 状态同步类会议预计减少 40-50% |
| 加速协作 | 跨团队信息获取从"等回复"变为"即时可得" |
| 保护核心人才 | 关键角色的可访问性提高，但注意力消耗降低 |
| 知识留存 | 分身交互记录形成组织知识沉淀 |
| 组织韧性 | 关键人不在时，分身仍能提供基础服务 |

## 五、总结

**Ask My Avatar First** 不是一个"更聪明的聊天机器人"。

它在做一件更基础、更深远的事情：

> **把一个人在工作中可代理的沟通层，重构为一个有状态感知、有权限控制、有升级决策能力的数字接口。**

它不替本人做所有事——它替本人**先接住世界**。

在一个人人都在用 AI 提高效率的时代，我们选择解决一个更本质的问题：

### 🛡️ 保护人类最稀缺的资源——不被打断的专注时间。

---

> **Let them talk to my avatar first.**
> **让别人先接触我的分身，而不是先打断我本人。**
