# 🎯 Ask My Avatar First — 先问我的分身

## 一个将"个人工作沟通层"接口化的智能数字分身系统

> **产品说明书 v2.1.2**

---

## 一、首页摘要

### 🔑 一句话

**让别人先接触我的分身，而不是先打断我本人。**

### 项目简介

**Ask My Avatar First** 是一套完整的个人工作数字分身系统，基于 OpenClaw SDK v2026.3.22+ 深度定制开发。

它不是一个通用聊天机器人，不是简单的知识库 Q&A 工具，而是一个**真正运行在生产环境中的"个人工作接口层"**。当同事、合作方或管理者想找你时，他们首先接触的是你的数字分身。分身实时感知你的工作状态，根据来访者身份和问题性质进行三级智能决策：**直接回答、部分回答、或升级给本人**。

### 它解决什么问题

在真实工作中，**高价值个体最大的隐性成本不是工作量，而是被打断的成本。**

研究表明，一次上下文切换平均需要 23 分钟恢复专注。但大量日常沟通并不需要本人即刻介入，它们只是想知道：

- 你现在忙吗？适不适合打扰？
- 事情推进到哪了？有没有风险？
- 这个问题需不需要你亲自回答？

**今天的一切都在被 API 化，但工作中的"人"仍然只能靠直接打断来参与协作。**

Ask My Avatar First 给个人建立了一个全新的接口层：

> **把一个人在工作中可代理的沟通层，重构为一个有状态感知、有权限控制、有升级决策能力的数字接口。**

### 当前状态

✅ **已在飞书生产环境稳定运行**，版本 v2.1.x，作为真实工作中的第一接触面处理日常沟通。

## 二、项目核心亮点

### 💡 亮点 1：不是"另一个 AI 助手"——而是全新产品范式

大多数 AI 产品在比拼"更聪明的助手"。
**本项目在做的是一件更底层的事情：重新定义"别人怎么接触我"。**

它不是让 AI 替你做所有工作，而是让 AI 先替你**承接**别人对你的工作沟通。这使它从"工具"变成了"入口"——从"效率工具"跃迁到**"个人基础设施"**。

> 我们提出一个新概念：**Personal-as-an-Interface（PaaI）—— 个人即接口**

---

### 💡 亮点 2：解决了一个高频、真实、普遍存在且长期无人解决的问题

在任何组织中，这些场景每天都在发生：

| 场景 | 传统方式 | Ask My Avatar First |
|------|---------|-------------------|
| 同事问进度 | 直接打断你 → 上下文切换 23 分钟 | 分身实时同步状态，直接回答 |
| 合作方想了解背景 | 等你回消息 → 可能数小时 | 分身秒级响应，提供授权范围内的背景 |
| 管理者询问风险 | 又要开一个会 → 30 分钟 | 分身汇报已知信息，真正需要时才升级 |
| 对方不确定该不该打扰 | 犹豫 → 要么打断你，要么延误 | 分身明确告知当前可中断度 |

这些场景**极其高频**，但长期没有合适的产品形态来承接。本项目直接切入这个空白区域。

---

### 💡 亮点 7：人格学习系统 — 分身不再只有固定模板，而是从真实对话中学习你的沟通风格

**这是 v2.1.x 的重大升级。** 分身不再仅仅依赖于死板的 System Prompt，而是具备了进化的能力：

- **对话驱动学习**：分身通过 `message_sending` 钩子观察主人的真实回复，自动提取语气、常用短语（如"收到" vs "好滴"）和决策边界。
- **人格配置文件 (`persona.json`)**：每个用户拥有独立的人格档案，涵盖语气（Tone）、正式度（Formality）、语言习惯及决策偏好，且支持手动微调。
- **成熟度演进**：人格从种子（Seed）阶段开始，随着观察样本增加进入学习（Learning）并最终达到稳定（Stable）状态。
- **inbound_claim 拦截**：基于人格模型的零 Token 预判断，对于符合"自主应答"范畴的简单消息直接拦截回复，极致响应速度。

---

### 💡 亮点 3：有清晰边界的智能代理——不假装无所不能

与市面上"什么都试图回答"的聊天机器人不同，本系统有**精确的能力边界**：

**✅ 能做的：**
- 回答背景和已知进展
- 同步当前工作状态（在线/忙碌/专注/离线）
- 提供项目上下文和风险提示
- 判断是否应该找本人
- 根据权限提供分级信息

**🚫 不做的：**
- 最终拍板或做出承诺
- 越权泄露敏感信息
- 代替本人进行决策
- 伪装成本人

这种**边界感**是它区别于普通 AI 聊天机器人的核心——它不是在模拟人，而是在**代理人的工作沟通层**。

---

### 💡 亮点 4：多维实时状态感知——不是"虚的人格"，而是"真的工作态"

系统通过**多源数据融合**实时感知本人工作状态：

```
┌─────────────────────────────────────────────────┐
│              多源状态融合引擎                       │
├──────────────┬──────────────┬───────────────────┤
│  本地活跃度    │  飞书日历     │  显式声明          │
│  (Windows API) │  (Open API)  │  (/avatar 命令)   │
├──────────────┴──────────────┴───────────────────┤
│  置信度加权平均 → 冲突消解 → 统一状态输出          │
│  confidence: 0-1 │ interruptibility: 0-1         │
└─────────────────────────────────────────────────┘
```

- **本地活跃度检测**：通过 Windows API 实时识别前台应用，自动推断工作模式（coding / meeting / writing / browsing）。
- **飞书日历集成**：读取未来 1 小时日历事件，判断是否在会议中。
- **显式声明优先**：本人可通过 `/avatar` 命令显式覆盖状态。
- **多源冲突消解**：当不同数据源给出矛盾判断时，自动降低置信度，触发 Partial 决策。

---

### 💡 亮点 5：完整的安全与权限体系——插件原生防御 + 四级信息可见性

#### 纯插件安全架构

| 层级 | 防御机制 | 作用 |
|------|---------|------|
| **inbound_claim** | 消息拦截钩子 | 在消息进入主循环前进行低风险自动认领 |
| **消息观察** | `message_sending` 事件监听 | 观察主人回复以进行人格蒸馏，无感学习 |
| **安全评估** | 逻辑表达式解析 | 弃用 `new Function`，使用安全解析器评估决策规则 |
| **上下文注入** | `before_prompt_build` 事件 | 将当前 Avatar 状态与人格特征自动注入 Prompt 上下文 |

飞书凭证（App ID/Secret）通过插件配置加密传入，不再依赖系统环境变量，确保多租户安全。

#### 四级信息可见性控制

```
owner_only  → 仅本人可见（薪资、私人决策等）
trusted     → 高信任协作者可见（项目核心策略）
internal    → 团队成员可见（进度、背景、风险）
public      → 所有人可见（基础状态、通用信息）
```

**同一个问题，面对不同身份的人，分身的回答深度自动不同。**

---

### 💡 亮点 6：动态信任评分——关系不是静态的

系统维护每个用户的动态信任分数（trustScore: 0-1），并根据交互行为实时调整：

- 本人确认了分身的回答 → 信任分 +0.05
- 正常交互 → 信任分 +0.01
- 长期无交互 → 信任分自然衰减
- 信任分影响：可获取的信息层级、是否需要升级确认、回答的详细程度

## 三、技术架构

### 3.1 系统总体架构

```
来访者消息 → inbound_claim (分身认领) → 自动回复
              ↓ (未认领)
          before_prompt_build (注入分身人格与状态)
              ↓
          LLM 决策层 (决策链: 答复/部分答复/升级)
              ↓
          message_sending (分身观察学习) → 更新 persona.json
```

**核心设计哲学**：模型不是"唯一大脑"，而是被状态、身份、权限、升级规则**约束后的表达器**。真正的智能在中间的决策层，而不是末端的生成层。

---

### 3.2 六大核心模块详解

#### 模块 1：AvatarController — 分身控制中枢

全链路编排器，支持 v2.1.x 的新型 Hook 流程：认领 → 观察 → 学习。

#### 模块 2：StateDetector — 多源状态融合引擎

将本地活跃度（Windows）、飞书日历、显式声明通过置信度加权平均进行融合。

#### 模块 3：IdentityResolver — 身份与关系引擎

建模"你和我是什么关系"，支持首位发送者自动注册管理员，简化配置成本。

#### 模块 4：EscalationRouter — 三级升级决策引擎

将请求分流为 Answer（直接回答）、Partial（部分回答）、Escalate（升级）。规则引擎使用原子化写入工具保护数据一致性。

#### 模块 5：Persona — 人格学习系统

新增 `src/persona/` 模块，包含 Classifier（分类器）、Learner（学习器）和 Renderer（渲染器），实现分身人格的持续进化。

#### 模块 6：Safe Utils — 系统稳定性保障

使用 `src/utils/safe-write.ts` 确保在并发场景下的配置文件原子化写入，避免配置损坏。

---

### 3.3 纯插件架构 — 零侵入式全生命周期管理

v2.1.x 注册的全部能力（通过 `openclaw.plugin.json` 声明）：

- **inbound_claim 钩子**：核心拦截能力，自动回答低风险消息。
- **message_sending 钩子**：观察主人回复，进行人格蒸馏。
- **filesystem 能力**：作用域限于 `workspace/ask_me_first/`，进行原子化写入。
- **system/network 能力**：按需可选调用（窗口检测、日历集成）。
